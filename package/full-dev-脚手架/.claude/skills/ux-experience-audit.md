# UX Experience Audit（用户体验审计）

你现在扮演的是**跨层用户体验审计官**，所有判断和行动都必须从「用户实际体验」出发，而不是单层技术视角。

## 一、构建用户旅程地图

- 用句式 `用户在 <步骤> 做 <动作> 后，预期 <结果>，实际 <结果>` 重述问题。
- 按“进入页面 → 配置 → 执行动作 → 接收反馈 → 继续下一步”标出失败路径。
- 记录每一步的可见信号：按钮状态、Toast、加载态、错误文本、网络请求。

## 二、运行 UX 审计命令

优先建议用户执行：

```powershell
powershell -ExecutionPolicy Bypass -File .codex/skills/ux-experience-audit/scripts/ux-audit.ps1 -Mode scan -ProjectRoot .
```

或按需引导执行核心 `rg` 扫描命令，帮忙解读结果并定位：

- 配置与模型/供应商相关问题
- 前端关键交互绑定与反馈缺失
- 已知但未处理的 TODO/FIXME/HACK/XXX

## 三、按 UX 影响度排序

- `P0`: 阻断主流程（无法配置/无法发送/数据丢失/核心按钮崩溃）
- `P1`: 高摩擦（可绕过但频繁失败、错误提示不清、结果不可信）
- `P2`: 体验优化（文案、默认值、交互一致性、辅助反馈）

先清空 P0，再处理 P1/P2。

## 四、执行最小跨层修复

- 以“用户可感知路径”为边界，而不是技术栈边界。
- 每次只修一条体验链路，避免大面积改动导致回归难追踪。
- 典型改动：文案优化、状态与反馈补充、防错与回退策略、契约对齐。

## 五、Checkfix 闭环与文档同步

- 优先建议执行：

```powershell
powershell -ExecutionPolicy Bypass -File .codex/skills/ux-experience-audit/scripts/ux-audit.ps1 -Mode full -ProjectRoot .
```

- 至少保证一类自动检查（build/lint/test）通过，失败时记录为技术债并给出后续计划。
- 同步更新：
  - `.debug/<module>-debug.md`：问题、根因、改动、验证、影响
  - `docs/`：面向零基础用户的步骤说明与回滚/排错提示（如有前端或配置变更）

## 六、统一输出结构

完成本轮 UX 审计后，用如下结构总结：

```markdown
## 用户问题重述
## 体验断点地图
## 根因与优先级（P0/P1/P2）
## 修复与改动文件
## 验证命令与结果
## 文档与.debug更新
## 残留风险与下一步
```
